# 从文件读写到Linux的VFS

## 普通的文件读写

平时看到的文件都是一长串数据。一般以字节为单位进行读写操作，操作步骤：

1. 调用系统提供的打开文件函数，获取文件的某种表示物
2. 通过这个表示物进行read/write/seek操作
    1. seek：改变光标位置，初始位置在文件开头
    2. read（简化）：指定从光标位置开始读取多少个字节，返回读取内容并移动光标
    3. write（简化）：和读取一样，换成从光标位置开始写入多少个字节
3. 使用完毕后可以关闭文件

## VFS理念

### “文件”的抽象

欸，你看read/write这俩函数，是不是可以理解为“接收/发送数据”？

那么，“读/写一个文件”换个说法就是“向文件发送数据/从文件接收数据”。（先不管seek）

所以我们可以把“一个文件”看成一个数据流接口，读/写操作其实是一种“数据传输”动作。

### 设备“文件”？

接下来试着把“文件”换成其他有数据流入流出的东西。比如我们有一个显示器，是不是可以用write来代表我们想往屏幕的像素点写入颜色信息？如果我们可以“向显示器写入”一大串图像数据，那么显示器拿到数据后就可以计算出所有RGB三色灯的电压，于是显示出你能看到的画面。所以，通过write函数向显示器“写入”数据是一种很自然的操作。

再来一个read的例子，比如我们的键盘，当我们按键时，电脑需要从键盘读取按键动作的数据。如果我们可以“从键盘读取”所有按键的状态数据，那么电脑就能够知道你按了什么键并做出相应的反应。所以，通过read函数从键盘“读取”数据也是一种很自然的操作。

### 你的下一个文件系统何必是“文件”系统？

回头看看我们的朴素文件系统模型（虽然只存在于我们的想象中），注意到有基本假设：数据以文件为单位组织，文件用文件夹来分类整理，然后所有数据按某种规则存放在硬盘这个巨型的约10^12个字节的列表里。

但既然我们现在已经把“文件”抽象成了一个数据流接口，有诸如显示器和键盘这样“设备文件”，那文件当然不可能对应到硬盘上的某一段空间了。

那就不对应了！移除定义“文件一定要对应到硬盘上的一段空间”，只保留树形的文件夹结构，毕竟这个分类方式还是很好用的。（此时这个文件树只是一个想象中的模型，没有落实到实际的存储介质上）

现在我们的树里有三种成员：

- `文件夹`（一个指向其它内容的指针列表）
- `普通文件`（数据）
- `设备文件`（设备连接时生成的id信息）

对于各种设备文件，读写它们时虽然做的确实是数据传输的操作，但实际的硬件原理肯定大不相同。我们希望使用统一的read/write接口来操作它们，那就肯定得有一套程序来把这些不同的设备文件的读写操作转换成对应的硬件操作。这些程序就是`驱动程序`，它们负责接收你发出的read/write等应用层的请求，进行处理后再真正地让硬件为你服务。

### 落实到硬盘

这个抽象的结构里的普通文件终究还是要存储到硬盘上。那我们可以这么设计：

- / `文件夹` 根目录，抽象概念
    - device/ `文件夹` 专门放设备文件
        - mouse1 `设备文件`
        - keyboard1 `设备文件`
        - disk1 `设备文件`
    - disk/ `文件夹` 为了“像以往一样访问硬盘”而存在
        - file1 `普通文件`
        - file2 `普通文件`

注意到上面的三个`文件夹`仅仅是因为它们包含了更多内容所以才叫文件夹，实际的职能并不同。

尤其是/disk，它是一个“需要实际存储”的文件夹。也就是说，当你操作里面的东西例如/disk/file1时，读写操作需要触发硬盘驱动程序，和硬盘产生交互，那么这个/disk就需要具有特殊的标记，表明对其中的内容进行读写操作时需要触发硬盘驱动程序。

### 挂载

隆重介绍：“挂载”！现在开始，文件夹可以被标记为“挂载点”，并附带称为“文件系统驱动”的程序（下称FS）。访问挂载点其中的内容时，会触发对应的FS，从而能和硬盘进行交互。

现在重新观察一下“访问多层文件夹”的动作，假设/a/b挂载了你的移动硬盘。当你试图访问/a/b/c时：

1. 从根目录(rootfs)/开始。其实根目录也是一个绑定了简单驱动的挂载点，名称是rootfs（对应sda等设备名），格式是tmpfs（对应fat32、ext4等文件系统）。毕竟硬盘对应的文件夹是个挂载点，和里面的内容交互要触发驱动，那根目录也套用这个模型就完事了，实际数据放在内存里即可（不多，放得下）。
2. 访问a。tmpfs回答有叫a的东西，当前对象是/a。
3. 访问b。tmpfs回答有叫b的东西，当前对象是/a/b。
4. 访问c。检测到/a/b文件夹有挂载点标记！也就是说它象征“包含更多文件的设备”，所以在访问其内容时，需要先切换到它绑定的FS，从而能和它挂载的硬盘进行交互。当前对象是(移动硬盘)/c。

现在我们可以更新一下上一个段落里的目录结构，哦对，文件夹还是改叫目录显得专业点：

- / `目录` rootfs挂载点
    - dev/ `目录` devtmpfs挂载点
        - disk1 `设备文件`
        - ...
    - disk/ `目录` 主硬盘挂载点
        - ...

还记得前面的设备文件吗？你这块硬盘当然也有一个设备文件，这个例子里它叫/dev/disk1。其实如果你真的精通文件系统（虽然不太可能）（笑），你甚至可以手动open这个/dev/disk1文件，然后seek到某个位置来读写数据。这和你在/disk/目录下直接读写文件，经由文件系统驱动向硬盘发送的信号是一样的。

### 文件操作

现在也可以进一步拆解“读写文件”动作了，假设你要写入/disk/file1：

1. 你调用open("/disk/file1")，VFS引擎开始解析路径：
    1. 从(rootfs)/开始，访问disk，存在，当前对象是/disk。
    2. 访问file1，但由于/disk是挂载点，先切换到它绑定的FS，当前对象是(主硬盘)/file1。
    3. 文件存在，你要打开它，创建一个`文件描述符`对象，登记了以下信息：
        1. 路径是/file1
        2. 光标在开头
        3. FS是主硬盘FS
            1. read函数地址
            2. write函数地址
            3. seek函数地址
            4. ……
        4. ……
    4. 把文件描述符作为open的返回值
2. 然后你调用的write就直接是主硬盘FS的write函数了，数据会直接写入主硬盘里边（如果你直接在(rootfs)/内创建文件，write就是tmpfs的write函数）

## 其他

上面的模型基本就是VFS的核心理念了，还有些不太重要的小事刚好想起来就顺便提提：

- 两种通信文件：`pipe文件`和`socket文件`，它们不是存储用的文件，只用于进程间通信。例如pipe文件就像一个管道，一头读一头写，可以用来实现分配任务给多个子进程执行等功能。
- 软链接文件：“看到我就去访问/a/b/c，至于它啥情况不关我事”
- 硬链接文件：“我和被链接对象没有区别，作为指针的内容完全相同”
- 你平时所看到的“目录树”仅仅是一个逻辑结构，其实VFS引擎在某种意义上只是一个“路径解析器”，当你进行一切“访问路径”的操作时，它就会从基于tmpfs的根目录开始，按你提供的路径中"/"之间的每一节字符串询问文件系统驱动并前进，中途遇到是挂载点的目录就切换到对应设备的FS，直到找到目标对象为止。
- 考虑到一些历史原因和性能问题，很多情况下"/"根目录上挂载的就是你的主硬盘，rootfs辅助系统启动后会被丢弃，系统也会在根目录里直接放几个需要持久保存的文件夹，例如/etc、/usr等，访问主硬盘里的文件也不需要每次都在前面加个/disk了。
